# コンテキストウィンドウとは何か — LLM が「見る」もの

> [前のページ](token-context-basics.md) で Token・Context・Context Window の基礎概念を学んだ。
> このページでは、Claude Code が Context Window の中に**具体的に何を配置するか**の全体像を見る。

## LLM の思考空間

LLM にとって、コンテキストウィンドウが「世界の全て」である。ウィンドウの外にある情報は存在しないのと同じ。

Claude Code では、この Context Window の中に以下が全て配置される:

## コンテキストウィンドウの構造

```
┌────────────────────────────────────────────────────────────────┐
│                    コンテキストウィンドウ                          │
│                                                                │
│  ┌──────────────────┐  常にロード（セッション開始時）               │
│  │ System Prompt    │  Claude Code内部のシステムプロンプト         │
│  │ (非公開)          │  ※ユーザーは変更不可                        │
│  └──────────────────┘                                          │
│  ┌──────────────────┐                                          │
│  │ CLAUDE.md        │  プロジェクト知識・規約（常時注入）            │
│  │ (階層マージ)       │  ※200行以内推奨                           │
│  └──────────────────┘                                          │
│  ┌──────────────────┐                                          │
│  │ Rules            │  条件付き注入（glob一致時のみ）              │
│  │ (.claude/rules/) │                                          │
│  └──────────────────┘                                          │
│  ┌──────────────────┐  オンデマンドロード                         │
│  │ Skills           │  （LLMが判断 or ユーザーが/で呼出）          │
│  │ (.claude/skills/)│                                          │
│  └──────────────────┘                                          │
│  ┌──────────────────┐                                          │
│  │ MCP Tools        │  ツール定義として注入                       │
│  │ (.mcp.json)      │  ※20kトークン超は性能劣化                   │
│  └──────────────────┘                                          │
│  ┌──────────────────┐                                          │
│  │ 会話履歴          │  ユーザーとの対話ログ                       │
│  │                  │  ※/compact で圧縮可能                     │
│  └──────────────────┘                                          │
│                                                                │
│  ┌──────────────────┐  コンテキスト外（LLMは見ない）               │
│  │ settings.json    │  ← ランタイム設定。LLMには注入されない        │
│  │ hooks            │  ← シェルで実行。LLMのコンテキスト外          │
│  └──────────────────┘                                          │
└────────────────────────────────────────────────────────────────┘
```

この図の最大のポイント: **settings.json と hooks は LLM のコンテキストウィンドウに入らない**。これらは Claude Code の「ランタイム」（Node.js に相当する部分）が処理するものであり、LLM の「思考」には直接影響しない。

## 「コンテキストに入る」と「コンテキスト外」の違い

| 区分               | 対象                                    | LLM への影響                           |
| :----------------- | :-------------------------------------- | :------------------------------------- |
| **コンテキスト内** | CLAUDE.md, Rules, Skills, MCP, 会話履歴 | LLM が直接「読んで」判断に使う         |
| **コンテキスト外** | settings.json, Hooks                    | LLM は存在を知らない。ランタイムが処理 |

この区分を理解することが、Part 3〜7 の設計判断を理解する鍵になる。

## 構造的問題との接続

Part 1 で学んだ構造的問題は、全てこのコンテキストウィンドウの制約に起因する:

- **Context Rot**: コンテキストが埋まるほど品質が低下
- **Lost in the Middle**: コンテキスト中間部への注意が低下
- **Priority Saturation**: コンテキスト内の指示が多いほど遵守率が低下
